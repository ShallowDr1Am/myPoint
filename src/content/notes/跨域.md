# 跨域解决方案

## 什么是跨域

浏览器的**同源策略**限制了不同源之间的资源访问。

同源需要满足三个条件：
- 协议相同
- 域名相同
- 端口相同

```
http://www.example.com/page1
http://www.example.com/page2     ✅ 同源

https://www.example.com/page1    ❌ 协议不同
http://api.example.com/data      ❌ 域名不同
http://www.example.com:8080      ❌ 端口不同
```

---

## CORS（跨域资源共享）

CORS 是 W3C 标准，允许浏览器向跨源服务器发送请求。

浏览器将 CORS 分为：**简单请求**和**非简单请求**。

### 简单请求

同时满足以下两个条件：

**1. 请求方法是以下三种之一：**
- GET
- POST
- HEAD

**2. HTTP 头部不超出以下字段：**
- Accept
- Accept-Language
- Content-Language
- Content-Type（仅限：`application/x-www-form-urlencoded`、`multipart/form-data`、`text/plain`）

### 简单请求流程

浏览器直接发送请求，自动在请求头中添加 `Origin` 字段：

```
GET /api/data HTTP/1.1
Host: api.example.com
Origin: http://www.example.com
```

服务器根据 `Origin` 决定是否允许：

```
// 允许的响应
HTTP/1.1 200 OK
Access-Control-Allow-Origin: http://www.example.com
Access-Control-Allow-Credentials: true
Content-Type: application/json

// 不允许的响应（无 CORS 头部）
HTTP/1.1 200 OK
Content-Type: application/json
```

### 非简单请求

不满足简单请求条件的请求，如：
- 请求方法为 `PUT`、`DELETE` 等
- `Content-Type` 为 `application/json`
- 自定义请求头

### 非简单请求流程

**第一步：预检请求（OPTIONS）**

```
OPTIONS /api/data HTTP/1.1
Origin: http://www.example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: X-Custom-Header
```

**第二步：服务器响应预检**

```
HTTP/1.1 200 OK
Access-Control-Allow-Origin: http://www.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: X-Custom-Header
Access-Control-Max-Age: 86400
```

**第三步：正式请求**

预检通过后，发送实际请求：

```
PUT /api/data HTTP/1.1
Origin: http://www.example.com
Content-Type: application/json
X-Custom-Header: value
```

### CORS 相关响应头

| 响应头 | 说明 |
|--------|------|
| `Access-Control-Allow-Origin` | 允许的源，`*` 表示所有 |
| `Access-Control-Allow-Methods` | 允许的请求方法 |
| `Access-Control-Allow-Headers` | 允许的请求头 |
| `Access-Control-Allow-Credentials` | 是否允许携带 Cookie |
| `Access-Control-Max-Age` | 预检请求缓存时间（秒） |
| `Access-Control-Expose-Headers` | 暴露给前端的响应头 |

### 携带 Cookie

**前端设置：**

```javascript
// XMLHttpRequest
const xhr = new XMLHttpRequest();
xhr.withCredentials = true;

// Fetch
fetch(url, {
  credentials: 'include'
});

// Axios
axios.get(url, { withCredentials: true });
```

**后端设置：**

```
Access-Control-Allow-Origin: http://www.example.com  // 不能为 *
Access-Control-Allow-Credentials: true
```

⚠️ **注意**：携带 Cookie 时，`Access-Control-Allow-Origin` 不能使用 `*`，必须指定具体域名。

## JSONP（基本没有使用场景）

利用 `<script>` 标签不受同源策略限制的特性。

### 实现原理

```javascript
function jsonp(url, callback) {
  const callbackName = `jsonp_${Date.now()}`;
  
  window[callbackName] = function(data) {
    callback(data);
    delete window[callbackName];
    document.body.removeChild(script);
  };
  
  const script = document.createElement('script');
  script.src = `${url}?callback=${callbackName}`;
  document.body.appendChild(script);
}

// 使用
jsonp('http://api.example.com/data', function(data) {
  console.log(data);
});
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 兼容性好 | 只支持 GET 请求 |
| 实现简单 | 存在安全风险（XSS） |
| | 无法检测请求是否成功 |
| | 需要服务器配合 |

## 代理服务器

通过服务器代理转发请求，绕过浏览器同源策略。

### 原理

```
浏览器 → 同源代理服务器 → 目标服务器
```

浏览器请求同源的代理服务器，代理服务器再请求目标服务器。

### Nginx 配置示例

```nginx
server {
  listen 80;
  server_name www.example.com;

  location /api/ {
    proxy_pass http://api.example.com/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
}
```

### Node.js 代理示例

```javascript
const express = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');

const app = express();

app.use('/api', createProxyMiddleware({
  target: 'http://api.example.com',
  changeOrigin: true
}));

app.listen(3000);
```

## postMessage

实现跨窗口通信，适用于：
- 页面与 iframe 通信
- 页面与 window.open 打开的页面通信

### 使用方法

**发送消息：**

```javascript
// 发送到指定源
otherWindow.postMessage('Hello', 'http://www.example.com');

// 广播（不推荐，存在安全风险）
otherWindow.postMessage('Hello', '*');
```

**接收消息：**

```javascript
window.addEventListener('message', function(event) {
  // 验证来源
  if (event.origin !== 'http://www.example.com') {
    return;
  }
  
  console.log('收到消息:', event.data);
});
```

### 安全注意事项

1. 发送时指定准确的 `targetOrigin`，避免使用 `*`
2. 接收时验证 `event.origin`
3. 验证 `event.source` 确保消息来源可信

## WebSocket

WebSocket 不受同源策略限制，支持双向通信。

### 使用示例

```javascript
const ws = new WebSocket('ws://api.example.com/socket');

ws.onopen = function() {
  ws.send('Hello Server');
};

ws.onmessage = function(event) {
  console.log('收到消息:', event.data);
};

ws.onerror = function(error) {
  console.log('连接错误:', error);
};

ws.onclose = function() {
  console.log('连接关闭');
};
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 不受同源策略限制 | 首次建立连接有握手开销 |
| 双向实时通信 | 需要服务器支持 |
| 数据格式灵活 | 兼容性需要考虑（旧浏览器） |

---

## SSE（Server-Sent Events）

服务器向客户端单向推送数据，基于 HTTP 协议。

### 使用示例

```javascript
const eventSource = new EventSource('http://api.example.com/events');

eventSource.onmessage = function(event) {
  console.log('收到消息:', event.data);
};

eventSource.onerror = function() {
  console.log('连接错误');
};

// 关闭连接
// eventSource.close();
```

### SSE vs WebSocket

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 单向（服务器→客户端） | 双向 |
| 协议 | HTTP | ws/wss |
| 重连 | 自动重连 | 需手动实现 |
| 数据格式 | 文本 | 文本/二进制 |
| 兼容性 | 更好 | 稍差 |

## 方案对比

| 方案 | 适用场景 | 限制 |
|------|----------|------|
| CORS | 常规 AJAX 请求 | 需要服务器配置 |
| JSONP | 简单 GET 请求 | 仅 GET，有安全风险 |
| 代理服务器 | 无法修改目标服务器配置 | 需要代理服务器 |
| postMessage | 跨窗口通信 | 需要目标窗口配合 |
| WebSocket | 实时双向通信 | 需要服务器支持 |
| SSE | 服务器推送 | 单向通信 |

---

## 总结

```
跨域解决方案选择：

需要常规请求？
├── 能修改服务器 → CORS
└── 不能修改服务器 → 代理服务器

只需要 GET？
└── JSONP（不推荐，有安全风险）

需要实时通信？
├── 双向 → WebSocket
└── 单向（服务器推送）→ SSE

跨窗口通信？
└── postMessage
```
