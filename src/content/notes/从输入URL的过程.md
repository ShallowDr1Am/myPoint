# 从输入URL到页面展示的过程

## 1. URL解析

浏览器首先解析输入的内容：判断是URL还是搜索关键词。如果是URL，分析协议（http/https）、主机名、端口、路径、查询参数等。

### URL编码（UTF-8）

URL只能包含ASCII字符（数字、字母和少数特殊符号），因此中文等非ASCII字符需要编码。

- **encodeURI**：编码整个URL，保留 `:/?#[]@` 等分隔符
- **encodeURIComponent**：编码URL参数值，会编码所有特殊字符（包括 `&=?#` 等），适用于拼接查询参数

## 2. DNS解析

将域名解析为IP地址，查找顺序：

1. 浏览器DNS缓存
2. 操作系统hosts文件
3. 本地DNS解析器缓存
4. 本地DNS服务器（递归查询）
5. 根DNS服务器 → 顶级域DNS服务器 → 权威DNS服务器（迭代查询）

> 步骤1-4为递归模式（客户端只问一次，中间环节代为查询），步骤5为迭代模式（本地DNS服务器逐级查询）。

### 递归 + 迭代查询示例

以 `m.n.com` 访问 `a.b.com` 为例：

1. 主机向本地DNS服务器发送查询报文（**递归**）
2. 本地DNS服务器向根DNS服务器发送查询（**迭代**）
3. 根DNS识别 `.com` 前缀，返回 `.com` 顶级域DNS服务器IP列表
4. 本地DNS服务器向顶级域DNS服务器查询（**迭代**）
5. 顶级域DNS识别 `b.com` 前缀，返回 `b.com` 权威DNS服务器IP
6. 本地DNS服务器向权威DNS服务器查询（**迭代**）
7. 权威DNS服务器返回 `a.b.com` 的IP地址
8. 本地DNS服务器将结果返回给主机

![DNS递归迭代查询](https://cdn.nlark.com/yuque/0/2022/webp/23128433/1667473124838-9cadb1e1-ae6c-4c95-9bf0-b81753679c69.webp)

### 前端优化：DNS预解析

```html
<link rel="dns-prefetch" href="https://baidu.com">
```

提前解析域名，减少DNS查询时间。

## 3. TCP三次握手

建立可靠连接：

| 次数 | 方向 | 标志 | 说明 |
|------|------|------|------|
| 第一次 | 客户端 → 服务器 | SYN=1, seq=x | 客户端发起连接请求 |
| 第二次 | 服务器 → 客户端 | SYN=1, ACK=1, seq=y, ack=x+1 | 服务器确认并回发连接请求 |
| 第三次 | 客户端 → 服务器 | ACK=1, seq=x+1, ack=y+1 | 客户端确认服务器连接 |

> **为什么不能两次？** 两次握手下，服务器无法确认客户端收到了自己的SYN+ACK。若客户端之前发送的旧SYN因网络延迟才到达，服务器会误建连接浪费资源。

## 4. TLS握手（HTTPS）

如果使用HTTPS，在TCP连接建立后还需TLS握手。以下对比三种方式：

### TLS 1.2 — RSA密钥交换（4次）

```
客户端 → 服务器：client_random, TLS版本, 加密套件列表
服务器 → 客户端：server_random, 选定的加密套件, 数字证书（含公钥）
客户端 → 服务器：用公钥加密的 pre_random
服务器 → 客户端：确认完成
→ 双方用 client_random + server_random + pre_random 生成会话密钥
```

**缺陷**：不具备前向安全性——私钥一旦泄露，所有历史通信可被解密。

### TLS 1.2 — ECDHE密钥交换（4次）

```
客户端 → 服务器：client_random, TLS版本, 加密套件列表
服务器 → 客户端：server_random, 加密套件, 证书 + server_params + 签名
客户端 → 服务器：client_params（验证证书和签名后发送）
→ 双方通过 client_params + server_params 用ECDHE算法计算出 pre_random
→ 再结合 client_random + server_random 生成会话密钥
```

**优势**：具备前向安全性，即使私钥泄露也无法推算历史会话密钥。

### TLS 1.3（1-RTT）

```
客户端 → 服务器：client_random, client_params, TLS版本, 加密套件列表
服务器 → 客户端：server_random, server_params, 加密套件, 证书
→ 双方直接通过 ECDHE 计算出 pre_random，再生成会话密钥
```

**优化**：客户端在第一条消息就发送DH参数，将握手从2-RTT缩短为1-RTT，还支持0-RTT恢复。

## 5. 发送HTTP请求

连接建立后，浏览器发送HTTP请求。首次请求通常是HTML文件。

### 慢启动

TCP采用慢启动算法：初始发送约14KB数据，根据确认帧反馈逐步增加发送量，直到达到网络最大带宽。拥塞控制算法根据丢包情况动态调整发送速率。

## 6. 浏览器缓存

收到响应后，浏览器根据缓存策略决定是否复用本地资源：

![浏览器缓存决策流程](cache-flow.gif)

```
请求资源
  │
  ├─ 强缓存命中？ ─── 是 ──→ 直接使用本地缓存（from memory/disk cache）
  │     字段：Cache-Control（优先）、Expires
  │
  └─ 否 ──→ 向服务器发送协商缓存请求
        │
        ├─ 协商缓存命中？ ─── 是 ──→ 返回304，使用本地缓存
        │     字段：If-None-Match / ETag（优先）
        │           If-Modified-Since / Last-Modified
        │
        └─ 否 ──→ 服务器返回新资源（200）
```

### 强缓存

- **Cache-Control**（优先级最高）：`max-age=xxx` 指定缓存有效期；`no-cache` 跳过强缓存直接进入协商缓存
- **Expires**：HTTP/1.0字段，指定过期时间点（受客户端时钟影响，已不推荐）

### 协商缓存

- **ETag / If-None-Match**（优先）：基于资源内容生成唯一标识，更精确
- **Last-Modified / If-Modified-Since**：基于文件修改时间，精度为秒级

### 缓存查找顺序

内存 → 硬盘 → 网络请求。加载到的资源同时缓存到内存和硬盘。

### 启发式缓存

当响应没有显式缓存指令但包含 `Date` 和 `Last-Modified` 时，浏览器取 `(Date - Last-Modified) × 10%` 作为缓存时间。

## 7. 解析与渲染

浏览器收到HTML数据后，进入关键渲染路径：

![关键渲染路径](render-path.gif)

### 第一步：构建DOM树

将字节流 → 字符 → Token → 节点 → DOM树。DOM构建是增量的。

- 遇到非阻塞资源（如图片），继续解析并异步下载
- 遇到CSS文件，不阻塞HTML解析，但阻塞JS执行（JS可能查询CSS属性）
- 遇到 `<script>`（无async/defer），阻塞HTML解析

**预加载扫描器**：主线程解析DOM时，预加载扫描器提前发现并请求高优先级资源（CSS、JS、字体），使主解析器到达时资源可能已就绪。

### 第二步：构建CSSOM树

CSS不是增量处理的——CSS可以被后续规则覆盖，因此必须等全部CSS接收完毕才能构建CSSOM。CSS是渲染阻塞资源。

选择器性能：`.foo {}` 比 `.bar .foo {}` 更快，因为后者需要沿DOM树向上检查祖先匹配。但通常不值得为此优化。

### 第三步：构建渲染树

DOM + CSSOM → 渲染树。渲染树只包含需要显示的节点：

- `<head>` 及其子节点：不出现
- `display: none`：不出现
- `visibility: hidden`：出现（仍占空间）

### 第四步：布局（Layout）

计算渲染树中每个节点的几何信息（位置、大小）。

- 首次计算称为**布局**
- 后续重新计算称为**回流（Reflow）**
- 节点数越多布局越慢，批量更新DOM、避免频繁修改盒模型属性可减少回流

### 第五步：绘制（Paint）

将渲染树节点绘制为屏幕上的像素：

- 首次加载绘制整个屏幕
- 后续只重绘受影响的最小区域

### 优化渲染的关键路径

1. 减少关键资源数量：异步加载、延迟加载或移除非关键资源
2. 减小关键资源体积：压缩、代码分割
3. 优化加载顺序：提高关键资源优先级，缩短关键路径长度

## 8. TCP四次挥手

数据传输完毕后断开连接：

| 次数 | 方向 | 标志 | 说明 |
|------|------|------|------|
| 第一次 | 客户端 → 服务器 | FIN=1 | 客户端请求释放连接，进入 FIN_WAIT 状态 |
| 第二次 | 服务器 → 客户端 | ACK=1 | 服务器确认收到，进入 CLOSE_WAIT 状态（此时服务器仍可发送数据） |
| 第三次 | 服务器 → 客户端 | FIN=1 | 服务器数据发送完毕，请求释放连接，进入 LAST_ACK 状态 |
| 第四次 | 客户端 → 服务器 | ACK=1 | 客户端确认，进入 TIME_WAIT 状态（持续2MSL后进入CLOSED） |

> **为什么需要TIME_WAIT？** 等待2MSL确保最后的ACK能到达服务器；若服务器未收到ACK会重发FIN，客户端可以重新应答。同时也让本连接的所有延迟报文在网络中消失，避免影响新连接。

## 性能指标

| 指标 | 含义 |
|------|------|
| **TTFB**（Time to First Byte） | 从发起请求到收到第一个HTML字节的时间 |
| **TTI**（Time to Interactive） | 从第一个请求到页面可交互的时间 |
| **FMP**（First Meaningful Paint） | 首次有意义的绘制——页面主要内容首次出现在屏幕上 |
